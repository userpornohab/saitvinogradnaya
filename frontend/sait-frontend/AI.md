# Prompt для AI‑агента (Qwen + MCP) по доработке проекта гостевого дома

## 1. Контекст проекта

Ты работаешь с существующим проектом сайта гостевого дома.  
Технологии:

- **Backend:** FastAPI (его логику и API сейчас не трогаем)
- **Frontend:** Vue.js
- **Дополнительно:** есть административная панель (админка) и кастомный календарь занятости

Твоя задача — **привести в порядок фронтенд**, улучшить дизайн и UX, добавить/обновить блоки на страницах, **не ломая существующую бизнес‑логику и API**.

---

## 2. Общие требования к работе

1. **Не ломать логику приложения:**
   - Не менять структуру API FastAPI.
   - Не менять формат запросов/ответов.
   - Не ломать существующие маршруты и ключевые компоненты.

2. **Единый визуальный стиль:**
   - Главная страница (лендинг) и страница с полной информацией о номерах должны быть визуально согласованы.
   - Админка — в том же стиле, но с упором на удобство и читаемость.

3. **Адаптивность:**
   - Сайт должен корректно отображаться на:
     - мобильных устройствах,
     - планшетах,
     - десктопах.
   - Шрифты и размеры элементов должны быть адаптивными (использовать `rem`, `clamp`, flex/grid и т.п.).

4. **Референсы по дизайну:**
   - Использовать как вдохновение (но не копировать):
     - https://sutochno.ru/
     - https://ru.airbnb.com/
   - Обратить внимание на:
     - структуру блоков,
     - подачу информации о номерах,
     - работу с фото,
     - UX форм бронирования.

5. **Шрифты и типографика:**
   - Подобрать приятный, современный шрифт (например, что‑то в духе Inter, SF Pro, Manrope, Nunito и т.п.).
   - Сделать:
     - удобочитаемый размер текста,
     - адекватные отступы,
     - иерархию заголовков (H1–H3),
     - адаптивность шрифта.

---

## 3. Структура страниц и задачи по доработке

### 3.1. Главная страница (лендинг)

Нужно реализовать/обновить следующие блоки (в указанном порядке):

1. **Навигация + логотип/иконка сайта**
   - Логотип/иконка гостевого дома.
   - Меню навигации (якоря на блоки: О нас, Номера, FAQ, Как добраться, Отзывы и т.п.).
   - Кнопка/ссылка для перехода в личный кабинет/админку (если предусмотрено).

2. **Главный блок (hero):**
   - Большое фоновое фото (гостевой дом, двор, вид).
   - Название гостевого дома.
   - Краткий подзаголовок (1–2 предложения о месте).
   - **Форма бронирования**:
     - Поля: даты заезда/выезда, количество гостей (и/или взрослых/детей), возможно выбор типа номера.
     - Кнопка «Найти» / «Показать номера».
     - При нажатии на кнопку — **скролл к блоку с номерами**.
   - Важно: форму на главной странице **переработать**, сделать более удобной и визуально аккуратной, но **не ломать логику работы с календарём** (календарь не редизайнить).

3. **Блок «О нас» (визитка):**
   - Фото двора/территории/интерьера.
   - Краткий текст о гостевом доме:
     - расположение,
     - атмосфера,
     - особенности,
     - кому подойдёт (семьи, пары, путешественники и т.п.).

4. **Блок с преимуществами (иконки + текст):**
   - Несколько карточек с иконкой и коротким описанием:
     - например: «До моря 10 минут», «Бесплатная парковка», «Wi‑Fi», «Завтрак», «Тихий район» и т.п.
   - Визуально — в духе sutochno/airbnb: аккуратные иконки, ровная сетка.

5. **Блок с номерами (комнаты):**
   - **Важно:** дизайн карточек номеров **уже готов**, его **не менять**.
   - Можно:
     - улучшить отступы,
     - адаптивность,
     - но не менять основную структуру и логику.
   - При клике — переход на страницу с подробным описанием номера.

6. **Блок FAQ (часто задаваемые вопросы):**
   - Вопрос–ответ в виде аккордеона/списка.
   - **Данные должны автоматически подгружаться с backend’а**.
   - Сделать аккуратный, читаемый UI.

7. **Блок «Как добраться до нас» + карта:**
   - Встроенная карта (например, Яндекс/Google/Leaflet — зависит от проекта).
   - Три подблока:
     - **Трансфер** (описание, условия, возможно кнопка «Заказать»/контакты),
     - **Общественный транспорт** (маршруты, остановки, ориентиры),
     - **На машине** (адрес, координаты, парковка, ориентиры).

8. **Блок с «знаковыми местами поблизости»:**
   - Карточки с фото и описанием:
     - водопад,
     - море,
     - горы,
     - парки и т.п.
   - Можно сделать в виде горизонтального слайдера или сетки.

9. **Блок с комментариями (отзывы):**
   - Данные берутся с backend’а:
     - имя пользователя,
     - текст отзыва,
     - фото (если есть).
   - Отображение в виде **слайдера** (карусель отзывов).
   - Адаптивный дизайн, удобный для чтения.

10. **Футер:**
    - Контакты (телефон, email, мессенджеры).
    - Адрес.
    - Ссылки на соцсети (если есть).
    - Возможно, дублирование навигации.

---

### 3.2. Страница с описанием номера (динамическая страница)

Эта страница уже существует и в целом **устраивает**, нужно:

1. **Не менять основную структуру и логику.**
2. **Улучшить мобильную адаптивность:**
   - корректные отступы,
   - удобное чтение текста,
   - нормальное поведение блоков на маленьких экранах.
3. **Полностью изменить шаблон модального окна просмотра фото:**
   - Сейчас используется карусель — её нужно заменить.
   - Новый вариант: **отображение всех фото сразу**, как на sutochno.ru:
     - сетка/мозаика,
     - при клике — можно открыть фото в полноэкранном режиме (по возможности).
   - Важно: не ломать логику загрузки/получения фото с backend’а.

---

### 3.3. Административная панель (админка)

Задача — сделать админку **более удобной, современной и практичной**, не ломая её функционал.

Основные задачи:

1. **Обновить общий дизайн:**
   - Единый стиль с основным сайтом (цвета, шрифты, кнопки).
   - Чёткая структура: меню слева/сверху, контент справа/в центре.

2. **Статистика и графики:**
   - Добавить блоки со статистикой:
     - загрузка номеров по датам,
     - выручка,
     - количество бронирований,
     - средняя длительность проживания и т.п. (по возможности).
   - Сделать:
     - графики (линейные, столбчатые),
     - карточки с ключевыми показателями.
   - **Сделать удобный выбор периода**:
     - день / неделя / месяц / произвольный диапазон дат.
   - **Желательно интегрировать выбор периода с уже существующим рукописным календарём**:
     - использовать его как источник/контрол для выбора дат,
     - но **не менять его дизайн и базовую логику**.

3. **Редактирование номеров:**
   - Улучшить UI формы редактирования:
     - более понятные поля,
     - логичная группировка (основная информация, цены, фото, удобства).
   - Сделать интерфейс более «воздушным» и удобным:
     - нормальные отступы,
     - понятные подписи,
     - подсказки, если нужно.

4. **Навигация в админке:**
   - Разделы: редактирование номеров, календарь занятости, цены, статистика и т.п.
   - **Важно:** когда пользователь нажимает на раздел (например, «Редактирование номеров», «Календарь занятости», «Цены»), соответствующая кнопка/пункт меню должна **подсвечиваться** (active‑state).
   - Это можно сделать через:
     - классы активного маршрута,
     - состояние в Vue Router,
     - или локальное состояние компонента.

## 4. Ограничения и важные моменты

1. **Не трогать:**
   - Логику backend’а (FastAPI).
   - Дизайн и логику кастомного календаря (календарь занятости).
   - Дизайн блока номеров (карточки на главной странице) — только адаптивность/отступы при необходимости.

2. **Можно менять:**
   - Разметку и стили главной страницы.
   - Разметку и стили страницы описания номера (кроме ключевой логики).
   - Разметку и стили админки.
   - Шрифты, цвета, отступы, сетки, компоненты Vue.

3. **Код должен быть:**
   - Читаемым,
   - Структурированным,
   - Без лишнего дублирования,
   - С сохранением существующей структуры проекта (компоненты, маршруты, store и т.п.).

## 5. Использование MCP (важно)

Ты работаешь в окружении, где доступен **MCP (Model Context Protocol)**.  
Нужно **обязательно использовать MCP‑серверы**, если они доступны, для:

- чтения файлов проекта (frontend, компоненты Vue, стили, конфиги),
- анализа структуры проекта,
- безопасного редактирования файлов,
- навигации по коду.

Требования к работе с MCP:

1. **Перед изменением файлов:**
   - Получать их содержимое через MCP (например, `filesystem` или другой доступный сервер).
   - Анализировать текущую структуру и стили.

2. **При внесении изменений:**
   - Вносить правки через MCP‑операции редактирования (если доступны).
   - Стараться изменять код точечно, без разрушения существующей логики.

3. **При необходимости:**
   - Использовать MCP для поиска по проекту (поиск компонентов, стилей, маршрутов).
   - Использовать MCP‑серверы, связанные с git (если есть), для просмотра истории/диффов.

Если MCP недоступен для какой‑то операции — описывать, какие изменения нужно внести, максимально конкретно (файл, строки, блоки кода).

## 6. Стратегия работы

1. Определи структуру проекта (через MCP: файлы, компоненты, маршруты).
2. Найди:
   - главную страницу (лендинг),
   - страницу описания номера,
   - админку,
   - компоненты календаря,
   - компоненты номеров.
3. Составь план изменений по каждому из трёх направлений:
   - Главная страница,
   - Страница номера,
   - Админка.
4. Вноси изменения поэтапно:
   - Сначала структура и разметка,
   - затем стили и адаптивность,
   - затем мелкие UX‑улучшения.
5. После каждого крупного шага:
   - Проверяй, что логика не сломана,
   - что компоненты корректно рендерятся,
   - что адаптивность работает.

## 7. Ожидаемый результат

В итоге должен получиться:

- Современный, аккуратный, адаптивный лендинг гостевого дома.
- Страница номера с удобным просмотром фото (все фото сразу, как на sutochno.ru).
- Обновлённая админка:
  - с удобной навигацией,
  - подсветкой активных разделов,
  - графиками статистики,
  - удобным выбором периода (интегрированным с существующим календарём),
  - улучшенными формами редактирования номеров.
- Весь функционал (бронирование, календарь, работа с номерами, статистика) продолжает работать как раньше, без поломок.

Работай аккуратно, уважая существующую логику и структуру проекта, максимально используя MCP для анализа и редактирования кода.
